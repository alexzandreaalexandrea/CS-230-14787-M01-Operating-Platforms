# CS-230-14787-M01-Operating-Platforms

Module 8 Journal

Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
Answer: The client was The Gaming Room and they wanted to expand their game Draw It or Lose It beyond Android and into a multi-platform, cloud-ready environment. The game needed to support multiple teams, multiple game instances, fast image rendering, and secure communication across different operating systems. 

What did you do particularly well in developing this documentation?
Answer: I think I did really well breaking down the technical requirements into clear sections and explaining them in a way that made sense for both developers and the client. The Recommendation section especially felt strong because it tied everything together, platform choice, architecture, storage, memory, distributed systems, and security. 

What about the process of working through a design document did you find helpful when developing the code?
Answer: Working through the design document first actually made the coding part easier. It forced me to think through the structure, the classes, the relationships, and the constraints before writing anything. By the time I got to the code, I already had a roadmap.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
Answer: If I could revise one part, I would update the Design Constraints section. I kept it pretty general the first time, and now I would add clearer details about performance limits, memory usage, and scalability. Making this section more specific would give the development team better guidance.

How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
Answer: The user needed a scalable, multi-platform game that was fast, secure, and easy to maintain. I focused on those needs when choosing Linux as the platform, recommending cloud storage, and outlining distributed communication. Considering user needs is important because the design has to solve their problems, not just be correct.

How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
Answer: My approach was to break everything down into smaller pieces, requirements, constraints, architecture, domain model, and recommendations. I tried to think about how each decision affects the others. In the future, I would use the same strategy, start with the problem, map out the system, and only then think about implementation. It keeps everything organized and prevents rework later.
